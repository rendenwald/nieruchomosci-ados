# CHANGE SPECIFICATION

> **PURPOSE**: Eliminate the TOCTOU race condition in `upsert_property()` by replacing the SELECT-then-INSERT/UPDATE pattern with an atomic `INSERT ... ON CONFLICT DO UPDATE` upsert.

## 1. SUMMARY

### 1.1 Problem

The current `upsert_property()` method in `services.py` uses a two-step pattern:

1. `SELECT` to check if a record exists by `(portal_source, source_id)`
2. If found → `UPDATE`; if not found → `INSERT`

This creates a **Time-of-Check-to-Time-of-Use (TOCTOU) race**: two concurrent callers can both `SELECT`, see no existing record, and both attempt `INSERT`. The unique constraint prevents data corruption, but one caller receives an `IntegrityError` instead of a clean result.

This was identified during the STORY-1 deep review (finding #1) and deferred. STORY-2 resolves it.

### 1.2 Solution

Replace the two-step pattern with an atomic upsert using PostgreSQL's `INSERT ... ON CONFLICT (portal_source, source_id) DO UPDATE` syntax. Keep the existing SELECT-then-INSERT/UPDATE pattern as a fallback for SQLite (used in tests).

This is a cross-dialect approach:
- **PostgreSQL**: Uses native `ON CONFLICT DO UPDATE` — handles ID generation via `GENERATED BY DEFAULT AS IDENTITY`
- **SQLite**: Uses the existing `max(id)+1` fallback with SELECT-then-INSERT/UPDATE (no concurrent test risk since SQLite tests are single-threaded)

## 2. TECHNICAL APPROACH

### 2.1 Current Implementation

```python
# services.py (current — simplified)
stmt = select(Property).where(
    Property.portal_source == validated.portal_source,
    Property.source_id == validated.source_id,
)
existing = (await self.session.execute(stmt)).scalar_one_or_none()

if existing is not None:
    # UPDATE existing record
    ...
    return existing, False

# INSERT new record — max(id)+1 fallback for SQLite
if "id" not in property_data:
    next_id = await self.session.scalar(
        select(coalesce(max(Property.id), 0) + 1)
    )
    property_data["id"] = next_id
new_property = Property(**property_data)
self.session.add(new_property)
await self.session.flush()
return new_property, True
```

### 2.3 Dialect Detection

Detect the database dialect once per service instance:

```python
class PropertyService:
    def __init__(self, session: AsyncSession) -> None:
        self.session = session
        self._dialect = session.bind.dialect.name if session.bind else "sqlite"
```

This avoids per-call overhead. The dialect is immutable for the lifetime of the service.

### 2.4 Cross-Dialect Comparison

| Concern | PostgreSQL | SQLite |
|---------|------------|--------|
| `ON CONFLICT DO UPDATE` | ✅ Native | ✅ Since 3.24.0 (2018) |
| `RETURNING` | ✅ Native | ✅ Since 3.35.0 (2021) |
| ID generation | `Identity(start=1)` | `max(id)+1` fallback |
| Concurrent test | ✅ Full isolation | ❌ StaticPool limitation |

### 2.5 Open Question

- **Returning the row**: After the upsert, we need to return the Property object with correct state. Options:
  1. Use `RETURNING` clause to get back all columns (PostgreSQL native; SQLite 3.35+)
  2. Re-SELECT after the upsert (works on all dialects but extra query)
  3. Use `session.refresh()` after execution (works but requires the object)

  **Recommendation**: Use `RETURNING` for PostgreSQL, `session.refresh()` for SQLite fallback.

## 4. RISKS AND MITIGATIONS

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| SQLite ON CONFLICT syntax differs from PostgreSQL | Low | Medium | Use distinct code paths per dialect |
| Session state corruption on failed INSERT | Low | High | `session.refresh()` after upsert clears stale state |
| RETURNING not supported on older SQLite | Low | Low | The CI/Dev SQLite will be 3.35+ (Ubuntu 22.04+) |
